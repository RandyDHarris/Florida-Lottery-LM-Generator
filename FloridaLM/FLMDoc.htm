Introduction

The FloridaLM application is a .net Windows Forms application that was developed using Object Oriented Programming (OOP) in the C# language. The main goal was to take a deeper dive into abstraction and follow the guiding principles that has been set forth in the industry. As it is difficult to follow and understand programming examples without knowing the end goal of a solution, this application is based on a real life scenario; picking lottery numbers.

 

The application links to the Florida Lottery website and retrieves the drawing history for the Lucky Money game by parsing the HTML formatted data and caching it. The application generates the top ten number sets (tickets to purchase) derived from an algorithm that utilizes the numbers most drawn in each position and when they are due to be drawn again. The definition of the algorithm is as follows: Query the numbers in each draw position and order them by most occurring for the last n draws. Each number is weighted by the longest period since the number was last drawn in each position. The number possibilities are from 1 to 47 for positions 1, 2, 3, and 4. The last position is the Lucky Ball. Since the numbers 47, 46, and 45 can never occur in position one and numbers 1, 2, and 3 can never occur in position 4, etc. the number of permutations are reduced which in fact reduces the total possibilities. Thus, there are 178,365 different ways in which 4 numbers can be chosen from a total of 47 unique numbers. The Lucky Ball may be between 1 and 17. The number of permutations increases substantively when including the Lucky Ball bringing the chance of actually guessing the correct combination to 1 in 3,032,205. It is important to note that when the algorithm generates numbers such that they are not ascending within each number set, those numbers are regenerated at random until the number set is a viable pick. After applying the algorithm to each position, the top ten values in each position are used to generate a set of 10 best tickets to purchase.

 

The Code

As mentioned in the introduction, this application follows the guiding principles found within the SOLID acronym and the ONION architecture. While there are many coding patterns that have been introduced to the industry such as the Singleton, Façade, and Mediator, the pattern found in this application aligns closest with the Singleton pattern.

 

SOLID

SOLID is an acronym that represents the first five principles. They are:

S

SRP

Single responsibility principle - a class should have only a single

 responsibility (i.e. only one potential change in the software's

 specification should be able to affect the specification of the class)

O

OCP

Open/closed principle - software entities should be open for extension, but closed for modification

L

LSP

Liskov substitution principle - objects in a program should be replaceable with instances of their subtypes without altering the correctness of that program

I

ISP

Interface segregation principle - many client-specific interfaces are better than one general-purpose interface

D

DIP

Dependency inversion principle - one should depend upon abstractions and not depend upon concretions

 

 

ONION

The ONION architecture has been widely adopted and when implemented typically ushers developers to follow the SOLID principals. Not all applications will follow all of the SOLID principles nor do they need to. It depends of course on the target solution, scope, and size of an application. ONION is not an acronym but represents a layering pattern as the onion has layers. This pattern allows developers to code and debug within layers and maintain separation of concerns. However, the key is to avoid dependency from one layer to the other correctly.

The application uses the Structuremap nuget package to inject dependency. When considering the reference dependency that the presentation project has on the Infrastructure project it is important to note that the reference is for dependency injection only. Additionally, there exist no dependencies on the presentation project at all. Hypothetically, the presentation could easily be removed from the solution without error. This could come in handy if the need arose to convert this solution to web based application. In fact, a new web project could be added and hooked up without removing the current project. That is the beauty of abstraction.

There is a test project included that uses mock data to perform unit testing. With this architecture, extensibility of the test project is infinite. Notice there are no dependencies or calls outside the Application and Business projects.
